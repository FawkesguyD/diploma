# ADR-0008: Разделение PostgreSQL на схемы `core` и `ops`

- **Дата:** 2026-05-14
- **Статус:** accepted

## Контекст

В PostgreSQL хранятся два разных по природе класса данных:

1. **Состояние системы** — что у нас есть и как настроено: пользователи, источники, реестр моделей, конфиги модулей.
2. **Журнал событий** — что в системе произошло: запуски парсера, запуски моделей.

Если складывать всё в одну схему `public`, теряется ментальная граница между «бизнес-данными» и «операционными записями». Когда позже захочется добавить observability-стек или вынести журналы в отдельную БД, придётся перетряхивать схему целиком.

## Решение

Разделяем таблицы PostgreSQL на **две схемы**:

### Схема `core` — состояние системы (бизнес-данные)
- `core.users`
- `core.sources`
- `core.model_registry`
- `core.module_configs`

### Схема `ops` — журнал событий / observability-light
- `ops.parser_jobs`
- `ops.model_runs`

FK между схемами разрешены и используются:
```sql
ops.model_runs.model_id        → core.model_registry.id
ops.model_runs.module_config_id → core.module_configs.id
ops.parser_jobs.source_id      → core.sources.id
```

Формулировка для защиты:
> *`core` — что система ЕСТЬ. `ops` — что в ней ПРОИСХОДИТ.*

## Альтернативы

- **Одна схема `public`** — проще на старте, но размывает границы; сложнее эволюционировать.
- **Префиксы имён** (`ops_model_runs`) — соглашение, а не структура; PostgreSQL-схемы — нативный механизм.
- **Две отдельные базы в одном Postgres** — теряем FK между схемами, лишние подключения.
- **Отдельный Postgres-инстанс для ops** — overkill для одного хоста ([ADR-0003](0003-single-host-docker-compose.md)).

## Последствия

**+** Чистая ментальная модель для защиты.
**+** Гранулярный доступ: можно дать read-only грант в `ops` будущему дашборд-сервису или Grafana, не открывая `core`.
**+** Лёгкая миграция позже: если заведём observability-БД, переносим только `ops`-схему.
**+** Бэкап-стратегии могут различаться: `core` критично, `ops` можно усечь по TTL.
**+** FK сохраняются через `ops.x.field REFERENCES core.y.id` — целостность не теряется.

**−** В коде сервисов нужно везде писать имя схемы (`core.users`, `ops.model_runs`). Решается `search_path` на роли или явными квалификаторами в SQLAlchemy-моделях.
**−** Alembic-миграции должны явно создавать схемы перед таблицами (`CREATE SCHEMA IF NOT EXISTS core; CREATE SCHEMA IF NOT EXISTS ops;`).

## Реализация

1. В начальной миграции Alembic:
   ```sql
   CREATE SCHEMA IF NOT EXISTS core;
   CREATE SCHEMA IF NOT EXISTS ops;
   ```
2. В SQLAlchemy-моделях: `__table_args__ = {"schema": "core"}` / `"ops"`.
3. В строке подключения можно задать `search_path=core,ops,public` для удобства интерактивных запросов, но **в коде всегда квалифицируем явно**.
