# Architecture Decision Records (ADR)

Здесь фиксируются **архитектурные решения** — что выбрано, почему, какие альтернативы рассматривались. Это страховка: если контекст потеряется (новая сессия, перерыв в работе, защита через полгода), ADR ответят на вопрос «почему мы сделали именно так».

## Формат

Каждое решение — отдельный файл `NNNN-краткое-название.md`:

```markdown
# ADR-NNNN: <название решения>

- **Дата:** YYYY-MM-DD
- **Статус:** proposed | accepted | superseded by ADR-XXXX
- **Контекст:** что заставило принять решение
- **Решение:** что выбрано
- **Альтернативы:** что рассматривали и почему отвергли
- **Последствия:** что это даёт, какие минусы, что нужно учитывать
```

## Когда писать ADR

- Выбор технологии/библиотеки/паттерна, который дальше будет влиять на код.
- Решение, которое нетривиально и его захочется объяснить позже.
- Любая «развилка», где один путь закрывает другой.

**НЕ нужно ADR** на тривиальное (порядок аргументов, форматирование, etc.).
