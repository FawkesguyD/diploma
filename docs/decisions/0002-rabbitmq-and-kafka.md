# ADR-0002: RabbitMQ для команд + отдельная Kafka для метрик

- **Дата:** 2026-05-14
- **Статус:** accepted

## Контекст

В системе два принципиально разных типа межсервисного взаимодействия:

1. **Оперативные команды** между сервисами: «спарси этот канал», «оцени эту пачку объявлений», «проклассифицируй сообщение». Низкая латентность, request/response-подобная семантика, не катастрофично потерять одно сообщение (можно перезапустить задачу).
2. **Поток метрик** для дашбордов: счётчики сообщений по темам, цены, активность по регионам. Объём — высокий, **потеря недопустима** (искажает аналитику и подрывает доверие к дашбордам), но допустимы задержки в секунды.

Эти два потока имеют разные требования к durability, throughput и backpressure. Смешивать их в одном брокере — значит оптимизировать под усреднённый профиль и проиграть на обоих.

## Решение

Два независимых брокера:

- **RabbitMQ** — для оперативных команд между сервисами (`parse.task`, `nlp.analyze`, `realestate.score`, `realestate.rank`). Лёгкие очереди, простая маршрутизация, удобный management UI.
- **Kafka** (отдельный кластер) — **только** для метрик (`metrics.messages`, `metrics.prices`, `metrics.system`). Append-only лог, гарантия доставки, ретеншн, возможность пере-чтения.

Каждый прикладной сервис (`realestate`, `nlp-parser`) пишет метрики в Kafka **независимо**, дополнительно к своей основной работе. Сервис `metrics` — единственный консьюмер Kafka, пишет в ClickHouse.

## Альтернативы

- **Только RabbitMQ** — потерялась бы гарантия доставки и удобный ретеншн для метрик. Метрика — это event, его естественно класть в лог.
- **Только Kafka** — overkill для оперативных команд, более сложная эксплуатация, нет удобного RPC-паттерна.
- **Прямая запись метрик из сервисов в ClickHouse** — связывает прикладные сервисы со схемой аналитической БД, делает невозможным rollout новой агрегации без изменения сервисов.

## Последствия

**+** Сбой/перегрузка Kafka не влияет на оперативный контур (RabbitMQ).
**+** Метрики можно перечитать (ретеншн), пересчитать агрегаты «с нуля».
**+** Прикладные сервисы не знают про схему ClickHouse — публикуют только события.

**−** Два брокера в инфраструктуре (двойная эксплуатация и память).
**−** Чуть больше сложности в коде сервисов: два клиента (`pika` + `aiokafka`/`confluent-kafka`).
