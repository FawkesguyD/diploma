# ADR-0004: MinIO как хранилище артефактов ML-моделей

- **Дата:** 2026-05-14
- **Статус:** accepted

## Контекст

Модуль оценки недвижимости использует CatBoost-модель, обученную в рамках статьи. В будущем добавятся NLP-модели (классификатор, NER, фильтр рекламы). Требования:

- модели нужно **версионировать** (модель v1, v2 — обе должны быть доступны для воспроизводимости результатов в дипломе);
- сервисы должны загружать конкретную версию по конфигу;
- бинарники модели не место в git;
- хочется не тащить внешний сервис вроде MLflow или DVC — это отдельная подсистема со своей сложностью.

## Решение

**MinIO** (S3-совместимое хранилище) как локальный объектный сторедж для артефактов.

Структура bucket:
```
models/
  realestate/
    v1.0/
      model.cbm
      metadata.json   ← фичи, гиперпараметры, метрики на отложке
    v1.1/
      ...
  nlp/
    classifier/
      v0.1/
        model.bin
        labels.json
```

Сервисы скачивают модель при старте (или по hot-reload), кешируют локально. Версия задаётся переменной окружения (`REALESTATE_MODEL_VERSION=v1.0`).

## Альтернативы

- **MLflow** — функциональнее (трекинг экспериментов, registry), но избыточен: нам нужно только хранить готовые артефакты, не вести эксперименты в его UI.
- **DVC + git-lfs** — DVC завязан на git-флоу, неудобно для рантайм-загрузки сервисом.
- **Хранить модели в образе Docker** — копируется при каждом релизе, нет версионирования отдельно от кода, образы раздуваются.
- **Хранить в Postgres / Mongo** — антипаттерн для бинарников, мусорит БД.

## Последствия

**+** Версионирование без MLflow.
**+** S3 API → стандартные клиенты (`boto3`, `minio`), не привязаны к конкретному сервису.
**+** Можно подменить MinIO на «настоящий» S3 (AWS / Yandex Object Storage) без изменения кода.
**+** Модели переживают пересоздание контейнеров (volume MinIO).

**−** Ещё один сервис в `docker-compose`.
**−** Метаданные моделей (фичи, метрики) приходится класть рядом в `metadata.json` — нет встроенного registry.
