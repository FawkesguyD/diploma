# ADR-0009: Redis как вспомогательный слой (dedup + outbound rate-limit)

- **Дата:** 2026-05-15
- **Статус:** proposed

## Контекст

Парсеры (`ingester` для Telegram/RSS, `realestate` для Avito/Cian) сталкиваются с двумя классическими проблемами распределённой системы:

1. **At-least-once семантика RabbitMQ.** Команда `parse` может быть доставлена воркеру повторно при ретрае (timeout consumer'а, рестарт брокера, NACK). Без идемпотентности — повторная HTTP-выборка из внешнего источника + повторная запись дублей в Mongo. Mongo upsert по `external_id` ловит дубли на уровне индекса, но это уже после совершения дорогих операций (HTTP, инференс).
2. **Outbound rate-limit к внешним источникам.** Avito/Cian/Telegram банят IP за превышение лимита. In-memory счётчик в воркере не работает при горизонтальном масштабировании (N реплик ingester'а — N независимых счётчиков). Нужен глобальный счётчик per-host, общий для всех реплик.

Дополнительно: при ретрае команды аннотации в RabbitMQ NLP-pipeline может прогнать ML-инференс дважды на одно и то же сообщение → лишняя нагрузка на CPU/GPU.

## Решение

Ввести **Redis 7** как вспомогательный слой для двух ролей:

1. **Dedup (идемпотентность):** атомарный `SET NX EX` с ключом `dedup:msg:{site}:{ext_id}` (TTL 24ч) перед дорогой обработкой. Аналогично `dedup:obj:*` для объявлений и `dedup:annotate:{msg_id}` / `dedup:eval:{obj_id}` (TTL 1ч) для защиты ML-инференса от повторного запуска.
2. **Outbound rate-limit:** счётчик `INCR ratelimit:src:{host}` + `EXPIRE 60` для sliding-window-приближения. При превышении лимита — `sleep(TTL)`, retry. Лимиты per-host из `infra/seed/source_limits.json`.

**Что Redis НЕ делает в рамках этого ADR:**
- не хранит бизнес-данные (сообщения, объекты, метрики);
- не заменяет RabbitMQ (durable очереди команд) или Kafka (event log метрик);
- не используется как cache дашбордов (отдельное решение, отложено);
- не используется как pub/sub для SSE (отдельное решение, отложено);
- не хранит JWT-сессии или blacklist токенов (JWT остаётся stateless).

**Конфигурация Redis:**
- `appendonly yes` + `appendfsync everysec` — durability с потерей ≤1 секунды при kill -9.
- `maxmemory 256mb` + `maxmemory-policy allkeys-lru` — жёсткий лимит, eviction старых ключей.
- `requirepass` — обязательная аутентификация.
- Один логический namespace (db=0), разделение через префиксы ключей.

## Альтернативы

| Альтернатива | Почему отвергнута |
|---|---|
| **Только Mongo upsert по `external_id`** | Дубли отсекаются после совершения HTTP-запроса и ML-инференса — дорогие операции уже выполнены. Не решает проблему rate-limit вообще. |
| **In-memory счётчик в воркере** | Не масштабируется горизонтально. При 2+ репликах ingester'а каждая держит свой счётчик → реальный outbound поток превышает лимит в N раз. |
| **Postgres-таблица `dedup_keys` с `INSERT ON CONFLICT DO NOTHING`** | Работает, но: (а) +1 IO в Postgres на каждое сообщение; (б) cleanup по TTL требует периодического `DELETE WHERE created_at < now() - interval '24 hour'` (фоновая задача); (в) в 5-10 раз медленнее Redis на коротких ключах. |
| **Postgres advisory locks для rate-limit** | Подходят для эксклюзивных локов, не для счётчиков. Для sliding-window нужна полноценная таблица + индексы. |
| **Полноценный sliding-window через Redis sorted sets (`ZADD` + `ZRANGEBYSCORE`)** | Точнее, но сложнее. Для защиты от ban'а погрешность ±60с на границе окна приемлема. Откладываем до появления реальной проблемы с точностью. |
| **Внешний rate-limit-сервис (например, Envoy/Traefik middleware)** | Работает только для inbound API. Для outbound к Avito нужен код в самом парсере. |
| **Полностью отказаться от защиты** | Гарантированный ban при тестировании парсеров; искажение метрик за счёт дублей в Kafka → CH. |

## Последствия

**Плюсы:**
- **+** Idempotency на стороне приёмника закрывает at-least-once брокера с минимальным overhead (~0.5мс на ключ).
- **+** Глобальный rate-limit работает для произвольного числа реплик ingester'а.
- **+** Защита ML-инференса от повторного запуска экономит CPU.
- **+** Graceful degrade: при недоступности Redis парсинг продолжает работать, дубли ловятся Mongo upsert (fail-open для dedup), лимиты заменяются на консервативный sleep 1с (fail-closed для rate-limit).
- **+** Готовый защитный аргумент на ВКР: «как обеспечивается exactly-once при ретраях», «как масштабируется outbound к внешним источникам».

**Минусы:**
- **−** +1 контейнер в `docker-compose.yml`, +1 зависимость в `services/_shared/`.
- **−** Дополнительная конфигурация (пароль, лимиты per-host) и мониторинг (memory, hit rate).
- **−** Окно неконсистентности при потере AOF (≤1 секунды) — теоретически дубли могут проскочить, но Mongo upsert ловит их вторым уровнем.
- **−** Sliding-window погрешность ±60с — приемлемо для защиты от ban'а, неприемлемо для billing-уровня точности.

**Что нужно учитывать дальше:**
- Если в будущем добавим cache дашбордов или pub/sub для SSE — это **отдельные ADR** (0010, 0011), которые расширяют использование того же Redis. Текущий ADR жёстко ограничивает скоуп двумя ролями.
- При горизонтальном масштабировании Redis (replica/cluster) — потребуется новый ADR. Сейчас single-node достаточно.
