# ADR-0006: Технологический стек

- **Дата:** 2026-05-14
- **Статус:** accepted

## Контекст

Нужно зафиксировать единый стек для всех сервисов, чтобы:

- не тратить время на выбор инструментов внутри каждого модуля;
- упростить переиспользование кода между сервисами (общие модели Pydantic, общие клиенты для брокеров и БД);
- унифицировать процессы сборки и деплоя (одинаковые `Dockerfile`-ы, типовые команды);
- использовать уже наработанный код из статьи ([model/](/Users/daniel/Projects/ДИПЛОМ/model) на Python + FastAPI).

## Решение

### Backend (все три прикладных сервиса + metrics)

- **Python 3.11+**
- **FastAPI** — HTTP API, OpenAPI «из коробки», совместимо с уже имеющимся кодом статьи.
- **Pydantic v2** — модели данных, валидация, сериализация. Единые модели на границах сервисов и между ними.
- **Uvicorn** — ASGI-сервер.
- **SQLAlchemy 2 + Alembic** — Postgres (уже используется в [model/alembic](/Users/daniel/Projects/ДИПЛОМ/model/alembic)).
- **Motor** / `pymongo` — MongoDB.
- **clickhouse-connect** — ClickHouse.
- **aio-pika** — RabbitMQ (async-клиент).
- **aiokafka** — Kafka producer/consumer.
- **minio** (официальный SDK) — артефакты моделей.
- **CatBoost** — модель из статьи.

### Frontend

- **Vue 3** (Composition API) + **TypeScript**.
- **Vite** — сборка и dev-сервер.
- **Node.js 20 LTS** — рантайм сборки.
- Раздача статики через nginx внутри Docker-образа frontend-сервиса.
- Библиотека графиков для дашбордов: ECharts или Chart.js (выбор отложен до проектирования дашбордов).

### Общее

- **Poetry** — управление зависимостями Python во всех сервисах (по образцу статьи).
- **pnpm** или `npm` — для frontend (TBD при создании сервиса).
- **Ruff** + **mypy** — линтер и типы для Python.
- **ESLint** + **Prettier** — для frontend.

## Альтернативы

- **Backend на Go / Node.js** — выше производительность сети, но потеряли бы интеграцию с CatBoost и пришлось бы переписывать наработки статьи. Не оправдано для дипломного объёма.
- **Django вместо FastAPI** — тяжелее, ORM не нужен для документного MongoDB-стека, OpenAPI требует допработ.
- **React вместо Vue** — равнозначно по возможностям; Vue выбран явно пользователем как предпочтительный.
- **Svelte / Solid** — экзотика для диплома, нет ощутимых плюсов.

## Последствия

**+** Один язык на бэкенде → переиспользование Pydantic-моделей между сервисами через общий пакет (`shared/schemas`).
**+** FastAPI генерирует OpenAPI автоматически → один из артефактов потока 1 (контракты) появляется «бесплатно».
**+** Vue + Vite — быстрый dev-цикл, лёгкая сборка в статику для раздачи через nginx.
**+** Можно напрямую портировать код из [model/](/Users/daniel/Projects/ДИПЛОМ/model) в `services/realestate`.

**−** Python не самый быстрый для high-throughput — для метрик ставим Kafka и батч-инсерт в ClickHouse, чтобы это не было узким местом.
**−** Нужно следить, чтобы версии Pydantic/FastAPI/SQLAlchemy совпадали между сервисами (фиксируем в общем lock-файле или базовом образе).
