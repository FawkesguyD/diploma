# ADR-0007: Все конфигурации системы — в PostgreSQL

- **Дата:** 2026-05-14
- **Статус:** accepted

## Контекст

Конфигурации аналитических модулей (какую модель использовать, пороги, batch size, расписания, тематики классификатора) можно хранить в нескольких местах:

1. **YAML/TOML в репозитории** + env-переменные.
2. **MongoDB** — рядом с другими документными данными.
3. **MinIO** рядом с артефактами моделей.
4. **PostgreSQL** — таблица `module_configs`.
5. **Etcd/Consul** — отдельный KV-стор для динамической конфигурации.

У каждого варианта свои плюсы (см. обсуждение), но смешивание «настройки» и «операционные данные» в одной БД (MongoDB) размывает границы и усложняет рассуждение об архитектуре.

## Решение

**Все конфигурации системы хранятся в PostgreSQL**, в таблице `module_configs` (плюс смежные таблицы `model_registry`, `sources`).

Чёткие границы хранилищ:

| Хранилище | Что в нём |
|---|---|
| **PostgreSQL** | Настройки и метаданные системы: пользователи, источники, конфиги модулей, реестр моделей, журналы запусков |
| **MongoDB** | Операционные данные пайплайна: сообщения, объявления, NLP-аннотации |
| **ClickHouse** | Аналитика и метрики для дашбордов |
| **MinIO** | Бинарники моделей (артефакты) |

Формулировка для защиты: *«PostgreSQL — всё, что про управление системой. MongoDB — всё, что система обрабатывает»*.

## Альтернативы

- **YAML в репозитории** — версионирование через git «бесплатно», но нельзя менять через UI/админку. Админка заявлена в плане.
- **MongoDB** — размывает границу «Mongo = операционные данные», усложняет ментальную модель.
- **MinIO + metadata.json** — годится только для параметров обучения модели, не для runtime-настроек модуля (расписания, источники).
- **Etcd/Consul** — лишняя инфра, нарушает [ADR-0003](0003-single-host-docker-compose.md) (минимум сервисов на один хост).

## Последствия

**+** Одна точка истины для всех настроек.
**+** Чёткая ментальная модель границ БД для всей команды и для защиты диплома.
**+** Транзакции, FK, ссылочная целостность для конфигов (важно: `module_configs.model_id → model_registry.id`).
**+** UI/админка пишет в одну БД, не нужно знать о нескольких источниках конфигурации.
**+** История изменений — через `audit_log` или через `updated_at` + версионирование (если потребуется).

**−** Изменение конфига требует SQL-миграции, если меняем структуру таблицы (не значений — значения в `params: jsonb`).
**−** Нет «бесплатной» истории изменений как в git — нужно добавлять `audit_log`, если потребуется.
