# ADR-0003: Один хост, развёртывание через docker compose

- **Дата:** 2026-05-14
- **Статус:** accepted

## Контекст

Это дипломный проект, а не production-система с SLA. Ключевые ограничения:

- ограниченное время на разработку;
- система должна **гарантированно стартовать** на стенде защиты одной командой;
- нужно показать архитектурную грамотность (микросервисы, очереди, разделение БД), но **не** заниматься DevOps бесконечно;
- никакого облака — всё локально, на одной машине.

## Решение

Вся система разворачивается на **одном хосте через `docker compose up -d`**.

- Корневой `docker-compose.yml` описывает все сервисы, БД, брокеры, шлюз.
- Две docker-сети: `edge` (публичная) и `internal` (приватная).
- Volumes для всех stateful-компонентов.
- Healthchecks на каждом сервисе.
- `taskfile.yml` оборачивает типовые команды (`task up`, `task down`, `task seed`, `task logs`).

## Альтернативы

- **Kubernetes (k3s/minikube)** — даст «серьёзности», но требует освоения кластера, ingress, манифестов. Время уйдёт на инфру вместо аналитической функциональности.
- **Запуск без контейнеров** (poetry/npm + системные сервисы) — невоспроизводимо у комиссии, легко сломать стенд.
- **Облачное развёртывание** — лишний барьер для защиты (нужен интернет, расходы, доступы).

## Последствия

**+** Один `docker compose up -d` поднимает весь стенд → защита проходит без сюрпризов.
**+** Легко переносится на любую машину с Docker.
**+** Микросервисная архитектура доказана, но без операционного оверхеда.

**−** Нет горизонтального масштабирования (для диплома неактуально).
**−** Нельзя «уронить» один сервис без эффекта на других при перегрузе хоста.
**−** Все секреты через `.env` — для production нужен был бы Vault/SealedSecrets, здесь — нет.
