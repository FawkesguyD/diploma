# Acceptance — критерии готовности к защите

> Чек-лист «что должно работать на дне защиты». Если пункт не закрыт — это блокер; если закрыт — есть демонстрация / артефакт. Стрелки на пункты других документов везде явные.

## Уровни готовности

- **MVP** (минимум для защиты) — без этого защищаться нельзя.
- **Target** (целевой) — то, что планируется довести; одного-двух пунктов из target можно не успеть без катастрофы.
- **Stretch** (бонусом) — успеем если останется время; пункты, которые сильно усиливают защиту, но не обязательны.

## MVP — обязательно

### Развёртывание

- [ ] `docker compose up -d` из корня репозитория поднимает весь стек без ручных шагов после клонирования (только заполнить `.env` из `.env.example`).
- [ ] Все сервисы healthy в `docker compose ps` в течение 60 секунд после старта.
- [ ] `curl http://localhost/` отдаёт SPA-фронт.
- [ ] `curl http://localhost/api/auth/login` отвечает 401 без токена (значит шлюз маршрутизирует).

### Авторизация

- [ ] Существует хотя бы один пользователь, созданный init-миграцией / seed-скриптом.
- [ ] Логин email+password возвращает JWT; запросы с истёкшим/невалидным токеном отдают 401.
- [ ] Роль `admin` имеет доступ к CRUD источников; роль `user` — нет.

### Парсинг

- [ ] Добавлен **хотя бы один Telegram-канал** и **хотя бы один новостной источник (RSS/сайт)** через UI или seed.
- [ ] Включённый источник опрашивается по расписанию автоматически (виден прирост в `messages`).
- [ ] Ручной запуск парсинга из UI приводит к появлению новых сообщений в ленте в течение 1 минуты.
- [ ] Сырое сообщение появляется в Mongo `messages` с правильными `channel_kind` и `channel_site`.
- [ ] Запуск отражён в `ops.parser_jobs` со статусом `succeeded`.

### NLP

- [ ] Для каждого нового сообщения автоматически создаётся документ в `annotated_messages` с `is_active: true`.
- [ ] Аннотация содержит непустые `topics`, `sentiment`, `is_ad` (даже если у модели нет уверенности — поля заполнены).
- [ ] Фильтр рекламы видимо работает: в ленте можно скрыть `is_ad=true` и количество карточек меняется.

### Недвижимость

- [ ] Подключён **хотя бы один источник объявлений** (cian.ru или avito.ru — достаточно одного).
- [ ] Объявления нормализуются в `objects` с заполненным `object_kind`, `channel_site`, `listing.address`.
- [ ] Модель CatBoost из MinIO загружается при старте `realestate`-сервиса.
- [ ] Для свежего объекта появляется `annotated_objects` с `predicted_price`, `deviation_pct`, `is_undervalued`.
- [ ] Эндпоинт `/api/objects/top-undervalued` возвращает непустой список с осмысленными отклонениями.

### Метрики и дашборды

- [ ] Сервис `metrics` подписан на оба Kafka-топика (`metrics.messages`, `metrics.prices`) и пишет в ClickHouse батчами.
- [ ] **Три обязательных дашборда** работают и показывают непустые данные (см. [dashboards.md](dashboards.md)):
  - [ ] активность по теме (например `badaevsky_complex`, 7 дней по часам);
  - [ ] динамика цен (хотя бы по одному городу + одному району);
  - [ ] топ недооценённых объектов (минимум 10 карточек).
- [ ] Дашборды читают MV/ad-hoc из ClickHouse через эндпоинты `/api/dashboards/*` — никаких хардкод-значений.

### E2E-сценарий «happy path»

- [ ] Сценарий 1: новое сообщение в источнике → появляется в ленте → имеет аннотацию → метрика отражена в дашборде «активность по теме».
- [ ] Сценарий 2: новое объявление → оценено моделью → попадает в `top-undervalued` (если `is_undervalued=true`) → видно в дашборде «динамика цен».
- [ ] Время от парсинга до видимости в UI ≤ 1 минуты на тестовом наборе.

### Документация

- [ ] `README.md` в корне репозитория с инструкцией «как запустить».
- [ ] `docs/design/*` заполнено (этот этап — готов).
- [ ] `docs/architecture/*.puml` собирается в PNG/SVG и вставлено в пояснительную записку.
- [ ] ADR-файлы зафиксированы для всех ключевых решений (`docs/decisions/`).

## Target — постараться

### Расширенный функционал

- [ ] Подписки пользователя (`core.user_subscriptions`): подписаться на тему / источник / объект, увидеть отфильтрованную «свою ленту».
- [ ] SSE-стрим `/api/messages/stream` работает — на фронте новые сообщения долетают без F5.
- [ ] Хотя бы 5 из 11 дашбордов из [dashboards.md](dashboards.md) (включая 3 обязательных).
- [ ] Карта районов с хороплет-заливкой по средней цене за м² (Leaflet + GeoJSON Москвы).

### Качество данных

- [ ] NER выделяет хотя бы три типа сущностей: location, developer, money — видно в карточке сообщения.
- [ ] Канонизация районов: `district_slug` корректно проставляется как минимум для топ-10 районов Москвы.
- [ ] Тематический классификатор: ≥ 10 тем в `labels.json` активной версии модели.

### Операционные качества

- [ ] DLQ для всех RabbitMQ-очередей; видно в management UI (`localhost:15672`).
- [ ] Все долгие операции возвращают 202+job_id; UI поллит и показывает прогресс.
- [ ] Логи всех сервисов в JSON-формате на stdout; видны через `docker compose logs`.

### Качество кода

- [ ] Все сервисы покрыты smoke-тестами на ключевые ручки/контракты.
- [ ] `mypy` / `pyright` без ошибок на бэкенде, `tsc --noEmit` без ошибок на фронте.
- [ ] Pre-commit с линтерами в репозитории.

## Stretch — бонусом

- [ ] Версионирование моделей: переключение активной версии CatBoost через UI, переоценка батча объектов под новую версию, сравнение «v1.0 vs v1.1» в дашборде качества модели.
- [ ] Сравнение двух тем рядом на графике активности (мультивыбор темы).
- [ ] Экспорт данных дашборда в CSV.
- [ ] Дашборд связей тем (co-occurrence граф).
- [ ] Граф «застройщик ↔ район» из NER-выгрузки.
- [ ] Алёрты в UI (звоночек): «по подписке появилось N новых сообщений / объект подешевел».
- [ ] Базовый аудит-лог админских действий (включил/выключил источник, переключил модель) — если решим всё-таки заводить `ops.audit_log`.

## Что отдельно проверяется на защите

Вопросы, к которым нужно быть готовым (= артефакты в репозитории должны это покрывать):

1. **«Почему три БД, а не одна?»** → [ADR-0001](../decisions/0001-three-databases.md).
2. **«Почему два брокера?»** → [ADR-0002](../decisions/0002-rabbitmq-and-kafka.md) + раздел «RabbitMQ vs Kafka» в [messaging/README.md](messaging/README.md).
3. **«Как версионируются модели?»** → [ADR-0004](../decisions/0004-minio-for-model-artifacts.md) + `core.model_registry` + `annotated_*` с `model_version`.
4. **«Почему недвижимость — модуль, а не ядро?»** → [ADR-0005](../decisions/0005-real-estate-as-module.md) + [scope.md](scope.md).
5. **«Как масштабировать?»** → реплики `*-worker` контейнеров; партиции Kafka можно поднять; для одного хоста ограничения.
6. **«Что будет если упадёт ClickHouse?»** → retention 7 дней в Kafka — пересоберём из лога.
7. **«Что будет если упадёт RabbitMQ?»** → durable очереди, `connect_robust` у воркеров; задачи переживут перезапуск брокера.
8. **«Как отслеживается качество модели?»** → дашборд `model-quality` поверх `mv_model_quality_daily`.

## Definition of done — формальная формулировка

Диплом готов к защите, когда:

1. Чек-лист **MVP** закрыт **полностью**.
2. Из чек-листа **Target** закрыто ≥ 70% пунктов.
3. Все ADR имеют статус `accepted`, ни одного `proposed`.
4. `docker compose up` на чистом хосте проходит без ручных правок (проверено хотя бы единожды).
5. E2E-сценарии 1 и 2 (см. выше) демонстрируются вживую без подготовки данных «прямо сейчас».
