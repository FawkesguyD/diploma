# Scope — объект, предмет, цели АИС

> Формальная «шапка» диплома: ровно то, что должно прозвучать на защите в первую минуту. Согласовано с [CONTEXT.md](../../CONTEXT.md), не противоречит теме и принятым ADR.

## Тема (фиксирована)

**Автоматизированная информационная система с интеллектуальным анализом и визуализацией данных.**

## Объект исследования

Информационные потоки и связанные с ними прикладные домены, в которых требуется автоматический сбор гетерогенных данных (тексты сообщений, структурированные объявления), их интеллектуальная обработка (классификация, NER, тональность, регрессионная оценка) и визуальное представление пользователю.

## Предмет исследования

Принципы, методы и архитектурные решения построения автоматизированной информационной системы, обеспечивающей:

- автоматический сбор данных из разнородных источников (Telegram-каналы, новостные сайты/RSS, площадки объявлений недвижимости);
- интеллектуальный анализ собранных данных (NLP-классификация, NER, тональность, фильтрация рекламы; регрессионная оценка стоимости и ранжирование объектов);
- хранение разнородных данных в специализированных БД (реляционное состояние, документное хранилище, аналитические агрегаты);
- визуализацию результатов через веб-интерфейс с настраиваемыми дашбордами.

## Цели АИС

1. **Автоматизировать** сбор информации из источников, выбранных и сконфигурированных пользователем-администратором.
2. **Применять** к собранным данным интеллектуальные методы анализа: NLP-классификация тематик, извлечение именованных сущностей, оценка тональности, фильтрация нерелевантного контента, регрессионная оценка стоимости объектов.
3. **Представлять** результаты в виде ленты сообщений, карточек объектов и аналитических дашбордов, отвечающих на конкретные пользовательские вопросы (динамика темы во времени, активность по районам, недооценённые объекты, поведение модели).
4. **Поддерживать расширяемость** — добавление нового аналитического модуля или источника данных не должно требовать переработки ядра системы.

## Задачи дипломной работы

1. Проанализировать предметную область и существующие подходы к построению АИС с интеллектуальным анализом.
2. Сформулировать функциональные и нефункциональные требования к системе.
3. Спроектировать архитектуру: контейнерное разбиение, межсервисное взаимодействие (RabbitMQ для команд, Kafka для метрик), хранилища данных (PostgreSQL / MongoDB / ClickHouse / MinIO), маршрутизация (API-gateway).
4. Разработать схемы данных для трёх БД и контракты сообщений между сервисами.
5. Реализовать аналитические модули:
   - модуль парсинга и NLP-обработки сообщений;
   - модуль регрессионной оценки и ранжирования объектов недвижимости (как прикладной кейс из исследовательской статьи);
   - модуль агрегации и дашбордов.
6. Реализовать веб-интерфейс пользователя (лента, фильтры, дашборды).
7. Развернуть систему как единый стенд (`docker compose up`) и провести E2E-проверку.

## Связь со статьёй

Параллельная научная статья **«Выявление потенциально недооценённых объектов на рынке недвижимости»** (CatBoost-регрессия + ранжирование) интегрируется в диплом как **один из аналитических модулей** ([ADR-0005](../decisions/0005-real-estate-as-module.md)). Это:

- даёт диплому реальный, отлаженный модуль вместо учебного «hello world»;
- оставляет тему диплома неизменной — недвижимость это **прикладной домен** одного из модулей, а не предмет работы;
- демонстрирует расширяемость АИС: показано, как «внешний» исследовательский результат подключается через те же контракты, что и любой другой модуль.

## Что **не** входит в работу

- Промышленная отказоустойчивость (multi-host, репликация БД, failover) — система рассчитана на один хост ([ADR-0003](../decisions/0003-single-host-docker-compose.md)).
- Полноценная система мониторинга/observability (Prometheus / Loki / Grafana) — спроектирована возможность подключения, но в реализацию диплома не входит.
- Мобильное приложение, публичные интеграции, биллинг.
- Обучение собственных тяжёлых NLP-моделей с нуля — используются готовые модели (классификатор / NER / sentiment) поверх предобученных энкодеров.
- Работа с коммерческой недвижимостью / землёй / парковками на уровне модели оценки — модель в статье обучена на жилом сегменте; поля типа в схеме оставлены под будущее расширение.

## Терминология (минимальный словарь для защиты)

- **АИС** — автоматизированная информационная система.
- **Источник** (`core.sources`) — настроенный администратором канал/сайт/RSS, из которого парсер периодически забирает данные.
- **Сообщение** (`messages`) — единица контента из информационного источника (Telegram-пост, новость, RSS-item).
- **Объект** (`objects`) — единица контента из источника недвижимости (объявление о продаже).
- **Аннотация / оценка** (`annotated_messages`, `annotated_objects`) — результат прогона сырой записи через модель; версионируется, актуальная помечается `is_active`.
- **Запуск модели** (`ops.model_runs`) — журнальная запись о факте инференса (когда, какая модель, на каком конфиге, что обработали).
- **Дашборд** — страница UI, отвечающая на конкретный пользовательский вопрос про рынок (см. [dashboards.md](dashboards.md)).
- **Шлюз** — Traefik (или Nginx) в роли единой точки входа HTTP, маршрутизирует `/api/*` на бэкенд-сервисы, `/*` — на frontend.
