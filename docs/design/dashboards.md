# Дашборды — функциональная спецификация

> Что именно показывает UI на странице «Дашборды». Для каждого виджета — назначение, фильтры, источник данных (MV в ClickHouse), эндпоинт API.
>
> Все дашборды отвечают на вопросы про **состояние рынка недвижимости и информационного фона вокруг него**. Технические метрики системы (latency, ошибки сервисов) сюда не идут — это observability ([clickhouse.md](databases/clickhouse.md)).

## Принципы

- **Один виджет = один эндпоинт API = один MV** (или ad-hoc запрос). Никакой склейки нескольких источников на фронте — это работа сервиса `metrics`.
- **Период** — общий пресет для всей страницы (24h / 7d / 30d / 90d / custom), плюс per-widget переопределение если осмысленно.
- **Гранулярность** — выводится из периода автоматически (24h → hour, 7d → hour, 30d+ → day, 90d+ → week/month). Пользователь может переопределить.
- **Пустое состояние** — виджет всегда показывает «нет данных за период» вместо ошибки. Скрывать виджет нельзя — иначе пользователь думает, что страница «битая».
- **Кэш** — на стороне `metrics` (Redis или in-memory LRU), TTL = гранулярности / 2 (для часовых данных — 30 мин кэш). Дашборды не должны бить ClickHouse на каждый F5.

## Страница «Главная»

Сводка для пользователя: что нового, что интересного.

### Виджет 1.1 — Топ-10 недооценённых объектов

- **Цель:** главный «крючок» системы — сразу показать ценность.
- **Тип:** карточки с фото / адрес / цена / отклонение от модели в %.
- **Фильтр:** город (по умолчанию — Moscow).
- **Источник:** `GET /api/dashboards/objects/top-undervalued` → ad-hoc SQL поверх `events_prices` (без MV).
- **Сортировка:** `deviation_pct ASC` (самые недооценённые сверху).
- **Действие:** клик → карточка объекта.

### Виджет 1.2 — Активность рынка за период

- **Цель:** общая картина «что происходит».
- **Тип:** 4 KPI-плитки:
  - `Новых объектов: N` (с трендом vs предыдущий период)
  - `Из них недооценённых: N`
  - `Сообщений по рынку: N` (минус реклама)
  - `Средняя тональность: 0.42` (positive/neutral/negative с эмодзи)
- **Источник:** агрегаты из `mv_district_activity_daily` + `mv_messages_by_topic_hourly`.
- **Эндпоинт:** `GET /api/dashboards/overview?since=...&until=...`.

## Страница «Цены»

### Виджет 2.1 — Динамика средней цены за м²

- **Цель:** «куда движется рынок».
- **Тип:** линейный график, ось X — время, ось Y — ₽/м².
- **Фильтры:** город, район (мульти-выбор для сравнения нескольких районов на одном графике), гранулярность (day / week / month).
- **Источник:** `mv_prices_timeseries_daily` (через `avgMerge(avg_price_per_m2_state)`).
- **Эндпоинт:** `GET /api/dashboards/prices/timeseries`.

### Виджет 2.2 — Распределение цен по комнатности

- **Цель:** «сколько стоит 1-комнатная vs 2-комнатная в моём городе сейчас».
- **Тип:** box-plot (медиана + p25/p75/p90) по rooms (1, 2, 3, 4+).
- **Фильтры:** город, период (по умолчанию — последний месяц).
- **Источник:** `mv_price_distribution_by_rooms_monthly` (через `quantilesMerge`).
- **Эндпоинт:** `GET /api/dashboards/prices/distribution?city=...&month=...`.

### Виджет 2.3 — Цены по районам (карта/таблица)

- **Цель:** где дороже / где дешевле.
- **Тип:** хороплет-карта (если успеваем по дипломному сроку — иначе таблица с цветовой заливкой).
- **Метрика:** средняя цена за м² за последний месяц.
- **Источник:** `mv_prices_timeseries_daily`, агрегация по последнему месяцу.
- **Эндпоинт:** `GET /api/dashboards/prices/by-district?city=...&month=...`.

## Страница «Информационный фон»

### Виджет 3.1 — Активность по теме

- **Цель:** «что обсуждают» — главная демонстрация NLP-модуля.
- **Тип:** stacked area chart, ось X — время, ось Y — кол-во сообщений; стопка по `channel_kind` (tg / news / rss).
- **Фильтры:** тема (single-select из справочника тем), гранулярность.
- **Пример:** *«сообщения по теме `badaevsky_complex` за 7 дней по часам»* — конкретный кейс из ТЗ.
- **Источник:** `mv_messages_by_topic_hourly`.
- **Эндпоинт:** `GET /api/dashboards/topics/activity`.

### Виджет 3.2 — Тональность по районам

- **Цель:** «где жители/СМИ пишут позитивно, где негативно».
- **Тип:** тепловая карта районов (или таблица): цвет от красного (negative) к зелёному (positive).
- **Дополнительно:** топ-3 негативных / позитивных района над картой.
- **Фильтры:** город, период.
- **Источник:** `mv_sentiment_by_district_daily`.
- **Эндпоинт:** `GET /api/dashboards/sentiment/by-district`.

### Виджет 3.3 — Связи тем (co-occurrence)

- **Цель:** «какие сюжеты идут парой» — академическая демонстрация работы классификатора.
- **Тип:** граф (узел = тема, толщина ребра = частота со-встречи) или матрица.
- **Фильтр:** период (по умолчанию — последний месяц).
- **Источник:** `mv_topic_cooccurrence_daily`.
- **Эндпоинт:** `GET /api/dashboards/topics/cooccurrence`.

## Страница «Качество модели»

> Дашборды поведения модели CatBoost. Полезно для защиты статьи и диплома: показать, что модель не «выдала числа и забыли», а отслеживается.

### Виджет 4.1 — MAE deviation_pct во времени

- **Цель:** «модель стабильна или рынок уплыл».
- **Тип:** линейный график с двумя сериями — средняя `|deviation_pct|` и p90.
- **Фильтр:** версия модели (single-select), период.
- **Источник:** `mv_model_quality_daily`.
- **Эндпоинт:** `GET /api/dashboards/model-quality`.

### Виджет 4.2 — Доля недооценённых по дням

- **Цель:** «сколько находок выдаёт модель в день» — sanity check ранжирования.
- **Тип:** столбчатая диаграмма.
- **Источник:** `mv_model_quality_daily` (поле `undervalued_state`) + `mv_district_activity_daily`.
- **Эндпоинт:** `GET /api/dashboards/model-quality/undervalued-share`.

## Страница «Активность по площадкам»

> Бизнес-картина предложения: где сколько объектов появляется.

### Виджет 5.1 — Поток новых объектов по площадкам

- **Цель:** «cian / avito / domclick — кто больше даёт, по типам недвижимости».
- **Тип:** stacked bar chart по дням, стопка по `channel_site`.
- **Фильтры:** тип недвижимости (`object_kind`), город.
- **Источник:** `mv_listings_by_channel_daily`.
- **Эндпоинт:** `GET /api/dashboards/listings/by-channel`.

## Минимум для защиты диплома

Если поджимает время — фокус на **трёх дашбордах**, которые прямо упомянуты в плане:

1. **Виджет 3.1** — «активность по теме *Бадаевского комплекса*» (NLP-модуль).
2. **Виджет 2.1** — «динамика цен» (рынок).
3. **Виджет 5.1** — «районы с лучшими покупками» — это пересечение **1.1 + 2.3** (топ объектов + цены по районам).

Остальные — расширение, если успеем по сроку.

## Решения по реализации

- **Карта районов — GeoJSON-подложка Москвы + Leaflet.** Хороплет-заливка по средней цене за м². Open-source GeoJSON границ районов берётся один раз (источник: московский Open Data или osm-boundaries) и кладётся как статика в `frontend/public/geo/moscow-districts.geojson`. Привязка района к данным — по `district_slug` (предварительная канонизация на стороне geocode-модуля). Без серверного рендеринга — всё на клиенте через Leaflet, бэк отдаёт только числа `district_slug → value`.
- **Экспорт данных (CSV/Excel) — не реализуем.** Не входит в scope диплома; если на защите спросят про выгрузку — ответ «дашборды доступны API-эндпоинтами JSON, выгрузка делается одной командой `curl … | jq` или подключением Excel/Power BI к OpenAPI». Реализация в коде — лишняя работа без осязаемой пользы.
