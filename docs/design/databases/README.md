# Схемы баз данных

В системе три БД, каждая под свой профиль нагрузки (см. [ADR-0001](../../decisions/0001-three-databases.md)). Этот раздел — целевые схемы, которые мы фиксируем перед началом разработки сервисов.

| Файл | Назначение | Что хранит |
|---|---|---|
| [postgres.md](postgres.md) | Реляционная БД | пользователи, источники, конфиги модулей, реестр моделей, журналы запусков |
| [mongo.md](mongo.md) | Документное хранилище | сообщения, NLP-аннотации, объявления недвижимости |
| [clickhouse.md](clickhouse.md) | Аналитическая БД | метрики, агрегаты для дашбордов, materialized views |

## Связи между БД

```
PostgreSQL              MongoDB                          ClickHouse
─────────────           ─────────────                    ─────────────
users                   messages ──────────────┐
sources ─────────────►  realestate_listings    │
model_registry ──────►  nlp_annotations        │
module_configs                                 │
model_runs ──────────►  (result_ref)           │
                                               ▼
                              события (Kafka) ──► events_messages
                                                  events_prices
                                                  events_system
                                                  ↓
                                              materialized views
                                              (агрегаты для дашбордов)
```

**Ссылочной целостности между БД нет** — связи поддерживаются на уровне приложения. Это сознательное упрощение: каждая БД — самодостаточна в своём слое.

## Принципы разделения данных

| Тип данных | Где | Почему |
|---|---|---|
| Пользователи, FK, конфиги | PostgreSQL | транзакции, ссылочная целостность |
| Сырые сообщения и объявления | MongoDB | гетерогенная структура, JSON-нативность |
| Метрики и агрегаты | ClickHouse | append-only, тяжёлые `GROUP BY` |
| Бинарники моделей | MinIO | версионирование артефактов |
